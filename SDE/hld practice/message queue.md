message queus are like a database but which serve some particular requirements, for example they allow the connection of the services asyncronously, they can have a producer and a consumer client, the producer only waits for the acknowledgement of the message queue, the good thing about message queues, is that they can guarantee that the recieved messages will arrive inthe order in which they were created by the producers,  if you wait for confirmation of the consumer, it can either slow down the system and then if you resend the data, then the ordering is not guaranteed, which means that we need to ensure that the current progress should be handled by the consumers, so that they can pick up from where they lost it, so that ordering can be maintained by the consumers, and the throughput can be high overall.
